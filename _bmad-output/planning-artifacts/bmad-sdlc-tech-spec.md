## 1. Overview

Extract the story automation pipeline from `who_else_is_here/automation/` (7 Python files, ~2,760 lines, 9 test files) into a standalone, pip-installable GitHub repo. The pipeline orchestrates Claude Code sessions through a 4-step SDLC workflow: **create-story -> dev-story -> code-review -> trace**.

### Value Proposition

Automate your Claude Code SDLC — from story creation through code review and traceability, with contract validation and audit trails.

### Current State

The pipeline is hardcoded to Who Else Is Here project paths, npm commands, BMAD skill names, Drizzle ORM checks, and vitest-specific test parsing.

## 5. Configuration Schema

Generated by `bsdlc init` at `.bsdlc/config.yaml`:

```yaml
project:
  root: .
  name: my-project

paths:
  sprint_status: _bmad-output/implementation-artifacts/sprint-status.yaml
  impl_artifacts: _bmad-output/implementation-artifacts
  planning_artifacts: _bmad-output/planning-artifacts
  test_artifacts: _bmad-output/test-artifacts
  runs: .bsdlc/runs

models:
  dev: opus
  review: sonnet

claude:
  bin: claude            # assumes claude is on PATH
  prompt_max_chars: 20000
  prompt_warning_chars: 15000

codex:
  bin: codex
  timeout: 600

build:
  command: npm run build
  timeout: 300

test:
  command: npx vitest run
  reporter_args: ["--reporter=json", "--outputFile={runs_dir}/test-results.json"]
  timeout: 300

timeouts:
  create-story: 600
  dev-story: 1200
  code-review: 900
  trace: 600

workflows:
  create-story: /bmad-bmm-create-story
  dev-story: /bmad-bmm-dev-story
  code-review: /bmad-bmm-code-review
  trace: /bmad-tea-testarch-trace

review:
  default_mode: A
  mode_b_tags: [security, auth, rbac, data-isolation]
  max_retries: 2
  extra_inference_keywords: {}
  # user additions merge on top of built-in INFERENCE_KEYWORD_MAP
  # example: "custom-auth-pattern": "auth"

safety:
  architectural_paths: ["*/schema/*", "*/migrations/*"]
  max_fix_files: 3

story:
  types: [scaffold, feature, refactor, bugfix]
  default_type: feature
  pipeline_steps: [create-story, dev-story, code-review, trace]

plugins: []
  # - drizzle_drift_check
```

### Variable Interpolation

Paths support `{project_root}` and `{runs_dir}` placeholders, resolved at config load time.

### `bsdlc init` Behavior

Interactive mode (default):
- Detects project type from package.json / pyproject.toml / go.mod
- Pre-fills build/test commands based on detection
- Prompts for model choices, workflow skill names, paths
- Generates `.bsdlc/config.yaml` and `.bsdlc/runs/`
- Appends `.bsdlc/runs/` to `.gitignore`

Non-interactive mode (`--non-interactive`):
- Writes config with all defaults, no prompts
- Suitable for CI bootstrapping

### `bsdlc validate`

Checks config + environment:
- Config YAML parses and passes schema validation
- Claude binary found on PATH
- Build command resolves
- Plugin entry points load

## 6. Configurable vs Core Behavior

| Configurable (user controls) | Not Configurable (core invariants) |
|---|---|
| Build/test commands | Step-based orchestration pattern |
| Workflow skill names | Contract validation logic |
| Model choices (dev, review) | Retry + escalation flow |
| Timeouts (per-step) | Mode A/B routing rules |
| Path layout | Run log schema |
| Plugin list | Safety heuristic logic |
| Extra inference keywords | Built-in inference keywords (INFERENCE_KEYWORD_MAP) |
| Architectural paths | StepStatus enum values |
| Story types | Ceremony/judgment step classification (STEP_MODES) |
| Mode B tags | ContractResult / RunResult data structures |

### Security-Critical Defaults

`INFERENCE_KEYWORD_MAP` and `MODE_B_TAGS` are hardcoded in the `Config` dataclass as safety invariants. The config key `review.extra_inference_keywords` **merges on top** of built-ins — it cannot remove or override them. This prevents accidental removal of `"csrf": "security"` or similar keywords that enforce Mode B reviews.

## 7. Plugin System

### Protocol

```python
# plugins.py
from dataclasses import dataclass
from typing import Protocol

@dataclass
class CheckResult:
    passed: bool
    message: str = ""

class PreReviewCheck(Protocol):
    name: str
    def run(self, story_key: str, config: "Config") -> CheckResult: ...
```

### Hook Point

The pipeline invokes `pre_review_checks` between dev-story verification and code-review. With no plugins configured, the pipeline runs identically to the current behavior minus the Drizzle drift check.

### Plugin Loading

1. Config lists plugin names under `plugins:`
2. Loader resolves via `importlib.metadata.entry_points(group="bmad_sdlc.plugins")`
3. Each plugin must implement `PreReviewCheck` protocol
4. Third-party plugins register via their own `pyproject.toml` entry points

### Bundled Plugin: `drizzle_drift_check`

- Runs `npx drizzle-kit push --dry-run` and parses output
- Returns `CheckResult(passed=False)` if drift detected
- Enabled by adding `drizzle_drift_check` to `plugins:` list

## 8. Key Refactoring Details

### auto_story.py -> orchestrator.py

- All `from config import X` becomes `from bmad_sdlc.config import get_config()` returning a frozen `Config` dataclass
- No module-level constants — all access via `Config` instance passed through function signatures
- `PROJECT_ROOT`, `SPRINT_STATUS`, `IMPL_ARTIFACTS` etc. come from `config.paths.*`
- `run_schema_drift_check()` call replaced by plugin hook invocation

### runner.py

- `["npm", "run", "build"]` replaced with `shlex.split(config.build.command)`
- `["npx", "vitest", "run", ...]` replaced with `shlex.split(config.test.command)` + `config.test.reporter_args`
- `CLAUDE_BIN` and `CODEX_BIN` sourced from `config.claude.bin` and `config.codex.bin`

### prompts.py

- All BMAD skill names (`/bmad-bmm-create-story` etc.) sourced from `config.workflows` dict
- `PLANNING_ARTIFACTS` path from `config.paths.planning_artifacts`
- `_bmad-output/implementation-artifacts/` in prompt templates replaced with `config.paths.impl_artifacts`

### state.py

- `STORY_TYPES`, `DEFAULT_STORY_TYPE` from `config.story.*`
- `INFERENCE_KEYWORD_MAP` from `Config._builtin_inference_keywords` merged with `config.review.extra_inference_keywords`

### contracts.py

- Already generic — no project-specific references. Confirmed by inspection. No changes needed beyond import path updates.

### run_log.py

- No changes needed. Already project-agnostic. Core invariant.

## 10. Preserved Core Patterns

These patterns are the soul of the pipeline and must survive extraction intact:

1. **Step-based orchestration**: create-story -> dev-story -> verify -> code-review -> trace
2. **Contract validation**: Each step has formal input/output contracts checked before proceeding
3. **Retry + escalation**: code-review -> dev-story -> code-review loop with max retries, then human escalation
4. **Run log audit trail**: YAML run log saved after every step for crash-safe auditability
5. **Mode A/B review routing**: Auto-selected by story tags, Mode B mandatory for security stories (no override)
6. **Safety heuristic**: [FIX] items touching architectural paths reclassified to [DESIGN]
7. **Pipeline-owned ceremony transitions**: Pipeline manages sprint-status, not the dev agent
8. **Independent verification**: Build/test run outside Claude session (AD-2)
9. **Context budget management**: Prompt size monitoring with truncation
10. **Resume/recovery**: `--resume` and `--resume-from` with run log reconstruction
